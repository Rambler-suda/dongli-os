# 1\. 产品概述

## 1\.1 产品一句话定义

**冻梨 OS** 是一个只属于你和女朋友的情侣生活小宇宙，用来记录在一起的时间、提醒彼此好好生活、保存未来想一起去的地方和想一起完成的事，并加入你们之间独有的“打赌筹码”小玩法。

它不是一个复杂的情侣空间，也不是健康管理 App，而是一个：

**实用为主、带纪念感、带一点专属玩笑和互动的小工具。**

更完整地说：

**冻梨 OS 是一个以 iOS 原生高级感为基础、以像素 IP 和像素小世界作为情绪表达层的情侣私密生活面板。**



---



## 1\.2 产品名称

主名称固定为：

冻梨 OS

英文副标题可使用：

DL universe

名称特征：

- 私人

- 可爱

- 不像普通情侣 App

- 有只属于你们自己的感觉

- 适合做成一周年礼物

- 具备后续扩展成“小宇宙”的想象空间

命名使用规范：

- App 内主标题统一使用“冻梨 OS”。

- Logo、启动页、首页可辅助展示“DL universe”。

- 不建议在界面中频繁使用“情侣 App”“恋爱空间”等泛化名称。



---



## 1\.3 产品定位

冻梨 OS 的定位不是“功能很多的成熟 App”，而是：

一个能每天打开看一眼的情侣轻量生活面板

一个把未来计划慢慢点亮的小系统

一个记录你们打赌小互动的地方

一个带有一周年礼物属性的私人小工具

它更接近：

- iOS 小组件式私人仪表盘

- 两个人的轻量生活提醒入口

- 旅行愿望与 Love List 的收藏盒

- 像素角色陪伴的小世界

- 打赌筹码的轻松记录面板



---



## 1\.4 产品不做什么

为了防止功能发散，V1 明确不做以下方向：

不是情侣社交 App

不是情侣相册

不是聊天软件

不是日记软件

不是健康管理工具

不是复杂任务管理工具

不是旅行攻略 App

不是备忘录替代品

不是成熟账号系统

不是大型游戏化养成系统

冻梨 OS 不追求功能全面，而追求：

她真的会打开

你们真的会用

这个东西真的像你送给她的

# 2\. 产品目标与优先级

## 2\.1 产品目标

V1 的核心目标有四个：

### 目标 1：每天打开有意义

用户打开 App 后，能立刻看到：

- 今天的分时问候

- 在一起天数

- 距离重要日期的天数

- 今日提醒 / 今日一句

- 两个人专属的小氛围

### 目标 2：随手记录未来计划

用户可以快速记录：

- 未来想一起去的地方

- 未来想一起完成的事

记录成本要低，不做复杂分类和流程。

### 目标 3：记录你们之间的小玩法

通过“筹码”页面记录你们打赌后的“小手余额”，让私密玩笑被认真又可爱地保存下来。

### 目标 4：形成独特风格记忆点

通过 iOS 高级感界面 \+ 像素头像/IP \+ 像素小世界点缀，让冻梨 OS 不像普通粉色情侣 App，也不变成低幼小游戏。

---



## 2\.2 产品优先级

冻梨 OS 的产品优先级为：

好用 \> 高级感 \> 纪念意义 \> 可爱感 / 像素感

解释如下：

### 第一优先级：好用

功能必须简单、直接、少层级。

用户打开后要能快速完成：

看今天提醒

看在一起多少天

添加旅游计划

添加 Love List

修改筹码余额

不需要每个功能都做得很深。

### 第二优先级：高级感

视觉必须精致、克制、清透。

整体以 iOS 原生高级感为基础，而不是普通粉色恋爱 App。

### 第三优先级：纪念意义

纪念感不靠堆页面，而靠关键细节：

在一起天数

双方头像 / 像素 IP

专属暗号 0607

专属名字“冻梨 OS”

旅游 LIST 和 Love List

打赌筹码玩法

分时问候和彩蛋文案

### 第四优先级：可爱感 / 像素感

可爱和像素风是加分项，不是主功能。

像素元素主要用于头像、IP、小世界、成就反馈、筹码图标，不影响主体功能效率。

# 6\. 首页需求

## 6\.1 首页定位

首页是冻梨 OS 的主面板，也是用户每天打开最先看到的页面。

它不承载复杂功能，只做三件事：

1. 显示在一起的时间

2. 根据当天时间给出生活提醒 / 问候

3. 给用户一个温柔、轻量、有专属感的状态入口

旅游、Love、筹码不要在首页展开。

首页最多只提供轻量快捷入口或状态提示，不把三个功能都塞进来。

---



## 6\.2 首页内容结构

首页从上到下建议为：

1. 顶部分时问候

2. 在一起天数主卡片

3. 生日 / 纪念日提醒卡片

4. 今日生活提醒卡片

5. 今日一句话

6. 轻量快捷入口

可在“在一起天数主卡片”或顶部区域加入轻量像素小世界背景，例如两个像素角色、一间小房间、一盏灯，但不单独做成复杂养成系统。

---



## 6\.3 顶部分时问候

### 6\.3\.1 展示逻辑

顶部问候采用“图标 \+ 主文案 \+ 副文案”的形式。

展示规则：

- 按本地时间判断当前时间段。

- 每个时间段有固定主文案。

- 副文案从对应文案池中随机选择。

- 建议每天同一时间段内副文案保持一致，避免频繁刷新造成跳动。

- 彩蛋文案可以保留，但应控制出现频率，避免过度打扰。

## 6\.4 在一起天数主卡片

### 6\.4\.1 固定日期

在一起日期固定为：

2025 年 6 月 7 日

不需要用户设置，不需要提供修改入口。

### 6\.4\.2 显示内容

主卡片显示：

我们已经在一起

XXX 天

辅助信息：

Since 2025\.06\.07

距离 X 周年纪念日还有 X 天

### 6\.4\.3 计算规则

- 在一起天数 = 当前日期 \- 2025\-06\-07 \+ 1。

- 周年纪念日 = 每年 06\-07。

- 如果当年 06\-07 已过，则计算到下一年 06\-07。

- 当天为 06\-07 时，显示周年祝福文案。

周年当天文案示例：

今天是我们的 X 周年

这一天，被冻梨 OS 认真收藏啦

### 6\.4\.4 视觉表现

- 该卡片为首页视觉重心。

- 采用大圆角白色 / 半透明白卡片。

- 可加入轻量像素小世界背景，例如两位小角色坐在房间里，但背景不能影响数字阅读。

- 数字“XXX 天”字号最大。

## 6\.5 生日提醒卡片

### 6\.5\.1 页面定位

生日提醒卡片用于在首页轻量展示双方生日距离天数，增加冻梨 OS 的个人化、纪念感和生活陪伴感。

它不是完整的纪念日管理功能，也不是生日礼物计划工具，只承担一个作用：

```Plain Text
提醒彼此的生日快到了。
```

卡片整体应保持小巧、安静、好看，不需要复杂交互。

---

### 6\.5\.2 固定生日数据

V1 中生日信息固定写入，不需要用户手动设置。

```Plain Text
我的生日：2002.02.03
婷婷生日：2002.03.18
```

页面展示时主要使用月日信息：

```Plain Text
我：02.03
婷婷：03.18
```

出生年份仅作为数据记录，不在首页卡片中重点展示，避免卡片信息过重。

---

### 6\.5\.3 卡片显示目标

生日卡片需要同时满足两个目标：

```Plain Text
1. 首页一眼能看到最近是谁的生日
2. 顺带看到两个人生日的距离状态
```

因此卡片不建议只显示一个人的生日，也不建议做成很大的倒计时模块。

推荐做法：

```Plain Text
主信息：距离最近生日还有 X 天
副信息：两个人生日的小标签
```

---

### 6\.5\.4 卡片布局建议

生日提醒卡片建议作为首页中的小卡片，尺寸低于“在一起天数主卡片”。

结构如下：

```Plain Text
┌──────────────────────┐

└──────────────────────┘
```

### 6\.5\.5 显示规则

系统每次进入首页时，自动计算距离当前日期最近的下一个生日。

计算逻辑：

```Plain Text
1. 分别计算距离“我的下一个生日”还有多少天
2. 计算距离“婷婷的下一个生日”还有多少天
3. 取天数更小的一方作为主提醒
```

如果今年生日已经过了，则自动计算下一年的生日。

示例：

```Plain Text
如果当前日期是 2026.01.20：
距离我的生日 2026.02.03 还有 14 天

如果当前日期是 2026.03.01：
距离婷婷生日 2026.03.18 还有 17 天

如果当前日期是 2026.06.07：
我的生日和婷婷生日都已过
则自动计算 2027 年的下一次生日
```

---

### 6\.5\.6 文案规则

生日卡片文案根据距离生日的天数变化，但不需要太复杂。

#### A\. 距离生日大于 30 天

```Plain Text
距离大冻梨生日还有 X 天
慢慢期待就好。
```

或：

```Plain Text
距离婷婷生日还有 X 天
先偷偷记上一笔小惊喜。
```

#### B\. 距离生日 8–30 天

```Plain Text
距离婷婷生日还有 X 天
要不要提前准备一点小惊喜？
```

或：

```Plain Text
距离大冻梨生日还有 X 天
这次可别忘记啦。
```

#### C\. 距离生日小于等于 7 天

```Plain Text
婷婷生日快到啦
还有 X 天，别装作不知道。
```

或：

```Plain Text
大冻梨生日快到啦
还有 X 天，可以开始期待了。
```

#### D\. 今天是生日

如果今天是婷婷生日：

```Plain Text
今天是婷婷生日
今天要被认真偏爱一下。
```

如果今天是我的生日：

```Plain Text
今天是大冻梨生日
今天可以理直气壮地快乐一下。
```

# 9\. 筹码页面需求

## 9\.1 页面定位

筹码页面是冻梨 OS 中最有个人特色、也最不像普通情侣 App 的功能。

它不是严肃记账，也不是惩罚系统，而是记录你们之间打赌后产生的“轻松小筹码”。

这个页面的核心不是：

谁欠谁多少

而是：

我们之间的小赌局，也可以被认真又可爱地记录下来。

---



## 9\.2 命名规则

页面主标题：

筹码

页面副标题可选：

愿赌服输，但只能轻轻来。

或：

这次先记上，不许赖账。

页面内部核心模块标题：

小手余额

界面中统一使用：

小手

小愿望

不建议频繁使用“巴掌”一词。

“巴掌”可以作为你们私下理解的含义，但界面表达统一为“小手”，这样更轻松、更可爱，也更符合冻梨 OS 的整体气质。

---



## 9\.3 页面核心概念

每个人都有一个自己的像素风虚拟形象。

页面结构可以理解为：

两个人的像素角色

\+

各自的小手余额

\+

快捷增减按钮

视觉上更像一个可爱的小游戏资产面板，而不是记账页面。

---

页面显示两张人物卡片，每个人一张。

每张卡片包含：

当前资产

大冻梨

⭐ ✋ ✋ ✋



\[\+1\] \[\+5\] \[\-1\] \[清零\]

筹码有两个层级：

也就是说：

推荐图标：

10 及以上：每 10 个小手合成 1 个小愿望

剩余不足 10 的部分继续显示小手

## 9\.12 页面最终定义

筹码页面最终定义为：

# 10\. 头像 / IP 形象系统

## 10\.1 系统定位

头像 / IP 不是普通资料展示，而是冻梨 OS 中最能体现“情侣专属感”和“像素小世界感”的核心视觉资产。

冻梨 OS 的 IP 不应被设计成泛化吉祥物，也不应只是两个抽象梨子角色，而应是基于你们两个人真实特征塑造的**双人像素角色**。

这套 IP 连接三个层面：

1. **身份识别**：让用户一眼知道“冻冻”对应男生，“梨梨”对应女生。

2. **情绪陪伴**：让 App 不只是工具，而像有两个属于你们的小角色住在里面。

3. **专属记忆**：将你们的外貌、身高、穿搭、爱好和关系特征转化为冻梨 OS 独有的视觉符号。

因此，冻梨 OS 的头像 / IP 系统最终应被定义为：

> **一套以真实情侣为原型、以柔和像素风呈现、服务于首页小世界与筹码玩法的双人角色系统。**

## 10\.2 双层头像系统

冻梨 OS 采用双层头像体系：

```Plain Text
真实头像层
像素 IP 层
```

### 10\.2\.1 真实头像层

真实头像用于基础身份识别。

适用场景：

```Plain Text
首次设置
个人资料识别
首页顶部小头像
设置 / 重新设置入口
```

视觉处理：

```Plain Text
圆形或大圆角裁切
统一尺寸
轻微白色描边
柔和阴影
避免直接大面积展示原图
```

真实头像的作用是“认出彼此”，不承担主要视觉风格。

---

### 10\.2\.2 像素 IP 层

像素 IP 是冻梨 OS 更重要的情绪表达层。

适用场景：

```Plain Text
首页小世界
筹码页面人物卡
完成 Love List 后的反馈
旅游 LIST 点亮后的旅行印章
空状态提示
隐藏彩蛋
未来成就系统
```

像素 IP 负责“情绪表达”和“专属记忆”。

它不是普通头像替代品，而是你们在冻梨 OS 小世界中的数字分身。

---

## 10\.3 IP 角色绑定关系

冻梨 OS 的双人 IP 明确绑定真实人物，不再采用“无性别泛化角色”。

```Plain Text
冻冻 = 男生 IP，对应“我”
梨梨 = 女生 IP，对应“女朋友”
```

两者应具有明确的性别气质、身高差、服装差异和性格特征。

角色关系不是“两个吉祥物”，而是：

```Plain Text
一个以男生为原型的冻冻
一个以女生为原型的梨梨
```

这能让 IP 更贴近真实人物，也更适合作为一周年礼物中的专属视觉记忆。

## 10\.4 像素 IP 总体风格

冻梨 OS 的像素 IP 不做传统 RPG 游戏角色，也不做低幼卡通头像。

```Plain Text

柔和像素风
低饱和色彩
小尺寸可识别
可爱但不幼稚
带一点真实人物特征
有情侣关系感
```

整体应遵循冻梨 OS 的视觉总原则：

```Plain Text
iOS 高级感为主
像素风作为情绪层
人物特征要清晰
不要做成低幼小游戏角色
不要做成战斗角色
不要过度卡通化
```

IP 应该像是住在 iOS 小组件里的两个小像素人，而不是游戏里的 NPC。

## 10\.5 角色一：冻冻

### 10\.5\.1 角色定位

冻冻是男生 IP，对应“我”。

角色气质：

```Plain Text
安静
冷静
清爽
有一点酷
有陪伴感
不夸张表达
偏理性但不冷漠
```

冻冻不应该被设计成“冰晶梨子吉祥物”，而应是一个带有真实人物特征的男生像素角色。

---

### 10\.5\.2 外貌特征

冻冻的核心外貌设定：

```Plain Text
性别：男生
身高感：高个子，约 180cm
头发：暗红色头发
肤色：微白，带一点小麦感，但整体仍偏白
体态：清爽、自然、略高挑
表情：淡淡的、轻松的、带一点不明显的笑意
```

视觉重点：

- 暗红色头发是冻冻最重要的识别点之一。

- 肤色不能画得太深，应是“偏白但带一点健康小麦感”。

- 身高要明显高于梨梨，但不要在小尺寸里做得夸张。

- 表情不需要太活泼，保持有点冷静、有点宠溺的感觉。

---

### 10\.5\.3 服装特征

冻冻的默认穿搭：

```Plain Text
黑色重磅短袖
白色宽松美式短裤
简洁休闲风
```

服装视觉要求：

- 黑色短袖要有“重磅 T 恤”的厚实感，不能画成普通紧身黑 T。

- 白色短裤应偏宽松、美式休闲，不要画成运动短裤或西装短裤。

- 整体穿搭应偏干净、利落、随性。

- 不需要过多配饰，避免复杂度过高。

冻冻的主视觉关键词：

```Plain Text
暗红头发
黑色重磅短袖
白色宽松美式短裤
高个子
清爽
轻酷
安静陪伴
```

---

### 10\.5\.4 像素表现建议

由于冻冻需要在小尺寸场景中使用，设计时应抓住几个最有识别度的特征：

```Plain Text
暗红发色
黑色上衣
白色短裤
较高的身高比例
偏冷静的表情
```

小尺寸下不需要复杂五官，只需要：

```Plain Text
简单眼睛
轻微嘴角
清晰发色
明确服装色块
```

## 10\.6 角色二：梨梨

### 10\.6\.1 角色定位

梨梨是女生 IP，对应“女朋友”。

角色气质：

```Plain Text
灵动
可爱
有活力
会跳舞
有一点酷甜感
亲密但不幼稚
```

梨梨不应只被设计成“柔粉色梨子角色”，而应是一个带有真实人物特征和舞蹈气质的女生像素角色。

---

### 10\.6\.2 外貌特征

梨梨的核心外貌设定：

```Plain Text
性别：女生
身高感：小个子，约 160cm
头发：深紫色头发
配饰：细框眼镜
体态：轻盈、灵动、有舞蹈感
表情：可爱、明亮、稍微活泼
```

视觉重点：

- 深紫色头发是梨梨最重要的识别点之一。

- 细框眼镜必须保留，这是区分梨梨的重要人物特征。

- 身高要明显低于冻冻，形成情侣身高差。

- 气质要更灵动、更活泼，体现爱跳舞的特征。

---

### 10\.6\.3 服装特征

梨梨的默认穿搭应围绕舞蹈气质设计。

着装方向：

```Plain Text
跳舞服装
Hiphop / Jazz 风格
轻运动感
有舞台感但不夸张
酷甜结合
```

可选服装元素：

```Plain Text
短款上衣
宽松舞裤
运动背心
短外套
宽松卫裤
舞蹈鞋
轻量配饰
```

服装视觉要求：

- 不做过度暴露或舞台化服装。

- 不做普通日常裙装，因为无法体现“爱跳舞”的特征。

- 可以偏 Hiphop / Jazz 练舞穿搭，体现灵动和力量感。

- 色彩以深紫、柔粉、奶白、浅灰、冻梨绿点缀为主，避免高饱和荧光色。

梨梨的主视觉关键词：

```Plain Text
深紫头发
细框眼镜
跳舞女孩
Hiphop / Jazz
160cm 小个子
灵动
酷甜
有活力
```

---

### 10\.6\.4 像素表现建议

梨梨在小尺寸像素形态中，应优先保留以下识别点：

```Plain Text
深紫发色
细框眼镜
舞蹈服装轮廓
较小身高比例
更活泼的姿态
```


视觉处理建议：

```Plain Text
冻冻高度约为梨梨的 1.10–1.15 倍
冻冻站姿更稳定
梨梨站姿更灵动
两人之间可以有轻微互动
```

推荐双人组合状态：

```Plain Text
并排站在小房间里
一起坐在沙发上

一起点亮 Love List
```

避免：

```Plain Text
两人比例完全一样
两人都像无性别吉祥物
两人服装和发色过于接近
动作过度夸张
画成低幼情侣贴纸
```

冻梨 OS 的 IP 不应直接画成两个梨子，而应以真实人物为主体，将“冻梨”概念作为细节融入。


```Plain Text

整体更柔软、灵动，呼应“梨”的亲和感
```

这样既能保留“冻梨 OS”的品牌概念，又不会把你们的角色做成普通吉祥物。

# 12\. 全局交互原则

## 12\.1 好用优先

用户打开 App 后应该立刻知道：

这个页面是干嘛的

当前状态是什么

我能点哪里

点完会发生什么

每个页面只解决一个主要问题：

首页：看今天和在一起天数

旅游：管理想去的地方

Love：管理想做的事

筹码：管理双方小手余额

---



## 12\.2 少层级

每个主功能最多两层：

主列表页

新增 / 编辑弹窗

不出现复杂三级页面。

例如：

旅游 → 新增地点弹窗

Love → 编辑事项弹窗

筹码 → 清零确认弹窗

---



## 12\.3 iOS 原生交互

推荐：

底部 Tab 固定切换

卡片点击反馈

胶囊按钮

底部弹窗 / 居中弹窗

轻量确认框

列表即时更新

输入框简洁清晰

避免：

复杂手势

隐藏很深的操作

长按才能发现的核心功能

过多浮层

过多二级菜单

---



## 12\.4 操作即时可见

用户完成操作后，结果必须立即反馈。

例如：

旅游地点点亮后，卡片马上变成“已去过”

Love 完成后，卡片马上变成“已完成”

筹码 \+1 后，小手马上增加

10 个小手后，马上合成为 1 个星星

头像上传后，页面马上显示新形象

推荐反馈方式：

卡片状态变化

小标签变化

轻微动效

短句提示

---



## 12\.5 删除必须确认

旅游和 Love 都有纪念意义，删除必须确认。

确认文案：

确定要删除这个想去的地方吗？

确定要删除这件想一起完成的小事吗？

确定要把小手余额清零吗？

删除完成后的反馈统一为：

已删除

---



## 12\.6 可逆操作优先

涉及状态变化的操作，尽量允许用户反悔。

旅游：点亮后可以取消点亮

Love：完成后可以取消完成

筹码：\+1 后可以 \-1

头像：可以重新设置

昵称：可以修改

# 17\. 给 AI / Codex / Cursor 的实现提示词

Create a mobile\-first PWA named “冻梨 OS” for a private couple use case\.



Core product:

- A lightweight couple life dashboard\.

- Four bottom tabs only: Home, Travel, Love, Chips\.

- First entry requires passcode “0607”, then setup two profiles with nicknames and avatars / pixel IPs\.

- Relationship start date is fixed: 2025\-06\-07\.

- Home shows time\-based greetings, relationship days, birthday / anniversary reminders, daily reminder, and daily quote\.

- Travel is a place\-based bucket list with add, edit, delete, mark as visited, undo visited\.

- Love is an experience\-based bucket list with add, edit, delete, mark as completed, undo completed\.

- Chips tracks two users’ hand balances with \+1, \+5, \-1, reset\. 10 hands automatically become 1 wish star\.

Visual style:

- Premium iOS native design is the base\.

- Use cream white, ice white, pear green, soft pink, ice blue, and mist purple\.

- Use large rounded cards, soft shadows, clean spacing, capsule buttons, and iOS\-like line icons\.

- Pixel art is only an emotion layer: avatars, mascot, tiny world, badges, hand/star feedback\.

- Avoid making the whole app look like a pixel game\.

- Cute details should be subtle, refined, and emotionally warm\.

Interaction:

- Keep all pages shallow and simple\.

- Use local storage first\.

- All operations should have immediate visual feedback\.

- Deletion and reset require confirmation\.

- Make it installable as a PWA on iPhone\.
