# 《冻梨 OS》交互规范 v1.0

文档类型：交互规范 / Interaction Guideline  
产品名称：冻梨 OS  
英文副标题：DL universe  
当前版本：V1.0 / 一周年礼物 MVP 版本  
适用对象：产品设计、UI 设计、前端开发、AI 辅助开发、Codex / Cursor 实现  

---

## 0. 规范定位

本交互规范用于统一冻梨 OS V1.0 的交互行为、页面反馈、状态变化、弹窗规则和页面层级。

冻梨 OS 是一个轻量情侣生活小系统，不是复杂任务管理工具，也不是大型社交产品。因此交互设计的核心目标不是“功能多”，而是：

```Plain Text
好理解
少层级
即时反馈
可反悔
不打扰
有一点专属情绪
```

最终交互体验应接近：

> 像 iOS 原生小工具一样清晰，像私人礼物一样有温度。

---

# 1. 全局交互原则

## 1.1 好用优先

冻梨 OS 的交互第一原则是：

```Plain Text
好用优先，不为氛围牺牲效率。
```

用户打开 App 后，应该立刻知道：

```Plain Text
当前页面是干嘛的
当前状态是什么
我可以点哪里
点完会发生什么
```

每个页面只解决一个主要问题：

| 页面 | 核心任务 |
|---|---|
| 首页 | 看今天状态、在一起天数、轻提醒 |
| 旅游 | 管理想一起去的地方 |
| Love | 管理想一起完成的事 |
| 筹码 | 管理双方小手余额 |

禁止把一个页面做成多个功能的大集合。

---

## 1.2 少层级原则

V1 所有主功能最多两层：

```Plain Text
主列表页
新增 / 编辑 / 确认弹窗
```

推荐结构：

```Plain Text
旅游 → 新增地点弹窗
旅游 → 编辑地点弹窗
Love → 新增事项弹窗
Love → 编辑事项弹窗
筹码 → 清零确认弹窗
```

不建议出现：

```Plain Text
列表 → 详情页 → 编辑页 → 确认页
```

冻梨 OS 是轻工具，不是复杂管理系统。

---

## 1.3 iOS 原生交互原则

交互方式应符合 iPhone 用户习惯。

推荐：

```Plain Text
底部 Tab 固定切换
卡片点击反馈
胶囊按钮
底部弹窗 / 居中弹窗
轻量确认框
列表即时更新
输入框简洁清晰
页面滑动自然
触控区域足够大
```

避免：

```Plain Text
复杂手势
隐藏很深的操作
长按才能发现核心功能
过多浮层
过多二级菜单
复杂路由跳转
```

核心操作必须可见，不依赖用户猜测。

---

## 1.4 操作即时可见

用户完成操作后，结果必须立即反馈。

例如：

```Plain Text
旅游地点点亮后，卡片马上变成“已去过”
Love 完成后，卡片马上变成“已完成”
筹码 +1 后，小手马上增加
10 个小手后，马上合成为 1 个小愿望
头像上传后，页面马上显示新形象
```

推荐反馈方式：

```Plain Text
卡片状态变化
小标签变化
轻微动效
短句 Toast
图标变化
```

反馈要明确，但不要打扰。

---

## 1.5 删除必须确认

旅游和 Love 都带有纪念意义，删除必须二次确认。

确认文案：

```Plain Text
确定要删除这个想去的地方吗？
确定要删除这件想一起完成的小事吗？
确定要把小手余额清零吗？
```

删除完成反馈统一为：

```Plain Text
已删除
```

不要在删除确认中使用过度拟人、油腻或恐吓式文案。

---

## 1.6 可逆操作优先

涉及状态变化的操作，尽量允许用户反悔。

必须支持：

```Plain Text
旅游：点亮后可以取消点亮
Love：完成后可以取消完成
筹码：+1 后可以 -1
头像：可以重新设置
昵称：可以修改
```

这样可以降低用户操作压力，也更符合轻量工具的使用习惯。

---

## 1.7 情绪反馈克制原则

冻梨 OS 需要有温度，但不能太吵。

适合的反馈：

```Plain Text
轻微发光
小角色轻跳
小手轻弹
星星轻闪
标签变亮
卡片柔和浮现
```

不适合的反馈：

```Plain Text
满屏撒花
大面积爱心
剧烈震动
复杂音效
强烈弹窗
持续动画
战斗特效
```

情绪反馈应该像“被轻轻回应了一下”，而不是“被打断”。

---

## 1.8 文案温柔但不油腻

文案气质应是：

```Plain Text
自然
亲密
简短
轻松
不命令
不过度撒娇
像真实会说的话
```

推荐表达：

```Plain Text
今天也一起完成一点点
这件小事，等你们一起点亮
又多了一段共同回忆
下一个想去的地方，选好了吗？
这里还空着，等你们一起填满
愿赌服输，但只能轻轻来
```

避免表达：

```Plain Text
公主请喝水
宝贝必须睡觉
不许反抗
必须执行
狠狠爱了
甜到爆炸
```

私人彩蛋文案可以保留，但主功能文案要克制。

# 2. 全局导航交互

## 2.1 底部导航结构

V1 底部导航固定为四个入口：

```Plain Text
首页 / 旅游 / Love / 筹码
```

不新增“回忆”“我们”等底部入口，避免第一版过重。

---

## 2.2 Tab 切换规则

底部 Tab 应始终可见，除首次进入流程外不隐藏。

交互规则：

```Plain Text
点击 Tab 立即切换页面
当前页面 Tab 高亮
未选中 Tab 弱化
切换动画轻微，不做复杂转场
切换后保留各页面当前状态
```

选中态建议：

```Plain Text
冻梨绿高亮
小胶囊背景
轻微图标放大
小像素点缀
```

禁止：

```Plain Text
像游戏菜单一样跳动
复杂 3D 转场
大面积颜色变化
Tab 入口数量超过 4 个
```

# 3. 首次进入流程交互

## 3.1 流程结构

首次进入流程为：

```Plain Text
打开冻梨 OS
↓
输入暗号 0607
↓
设置我们的形象
↓
进入首页
```

流程目标：

```Plain Text
有一点仪式感
但不能复杂
不能像账号注册
不能让用户填写太多东西
```

---

## 3.2 暗号页交互

暗号固定为：

```Plain Text
0607
```

对应在一起日期：

```Plain Text
2025.06.07
```

页面文案：

```Plain Text
进入冻梨 OS
猜猜暗号是什么
```

输入规则：

```Plain Text
输入 4 位数字
输入完成后自动校验，或点击确认校验
输入错误时不清空全部内容，可允许用户直接修改
```

错误提示：

```Plain Text
暗号好像不对，再想想。
```

输入正确后：

```Plain Text
进入“设置我们的形象”页面
记录首次进入已通过暗号
```

---

## 3.3 设置我们的形象交互

页面标题：

```Plain Text
设置我们的形象
```

副文案：

```Plain Text
先让冻梨 OS 认识你们
```

需要设置：

```Plain Text
我的昵称
我的头像 / 像素 IP
她的昵称
她的头像 / 像素 IP
```

默认角色：

```Plain Text
冻冻 = 男生 IP，对应“我”
梨梨 = 女生 IP，对应“女朋友”
```

交互规则：

```Plain Text
用户可上传真实头像
用户也可直接使用默认像素 IP
如未上传头像，自动使用默认像素 IP
设置完成后进入首页
后续可在重新设置入口修改
```

---

## 3.4 二次进入规则

首次设置完成后：

```Plain Text
再次打开 App 直接进入首页
不再要求输入暗号
保留重新设置资料入口
```

---

# 4. 首页交互规范

## 4.1 首页定位

首页是每天打开看一眼的主面板。

首页不承载复杂管理功能，只做：

```Plain Text
展示当前时间状态
展示在一起天数
展示生日 / 纪念日提醒
展示今日生活提醒
展示今日一句
提供轻量快捷入口
```

旅游、Love、筹码不要在首页展开完整列表。

---

## 4.2 分时问候交互

顶部问候采用：

```Plain Text
图标 + 主文案 + 副文案
```

展示规则：

```Plain Text
按本地时间判断当前时间段
每个时间段有固定主文案
副文案从对应文案池随机选择
建议同一天同一时间段内副文案保持一致
彩蛋文案控制出现频率，避免过度打扰
```

时间段：

```Plain Text
06:00–07:00 清晨过渡
07:00–09:00 早上
09:00–12:00 上午
12:00–15:00 中午
15:00–18:00 下午
18:00–21:00 傍晚
21:00–24:00 晚上
00:00–03:00 半夜
03:00–06:00 深夜
```

注意：

```Plain Text
时间判断使用本地时间
不要使用 UTC 导致时间段错位
文案刷新不要频繁跳动
```

---

## 4.3 在一起天数卡片交互

该卡片为首页视觉重心。

固定日期：

```Plain Text
2025.06.07
```

显示内容：

```Plain Text
我们已经在一起
XXX 天
Since 2025.06.07
距离 X 周年纪念日还有 X 天
```

计算规则：

```Plain Text
在一起天数 = 当前本地日期 - 2025-06-07 + 1
周年纪念日 = 每年 06-07
如果今年 06-07 已过，则计算到下一年 06-07
如果当天是 06-07，显示周年祝福文案
```

周年当天文案示例：

```Plain Text
今天是我们的 X 周年
这一天，被冻梨 OS 认真收藏啦
```

交互规则：

```Plain Text
V1 不做点击详情页
不提供修改在一起日期入口
卡片只做展示
```

---

## 4.4 生日提醒卡片交互

生日卡片用于轻量提醒双方生日。

固定生日：

```Plain Text
我的生日：2002.02.03
婷婷生日：2002.03.18
```

展示规则：

```Plain Text
每次进入首页时计算距离最近的下一个生日
显示最近生日对应的人和剩余天数
底部可展示两个人生日小标签
```

卡片结构：

```Plain Text
🎂 生日提醒
距离婷婷生日还有 X 天
要不要提前准备一点小惊喜？
大冻梨 02.03   婷婷 03.18
```

生日当天文案：

```Plain Text
今天是婷婷生日
今天要被认真偏爱一下。
```

V1 不支持：

```Plain Text
编辑生日
生日礼物清单
生日详情页
农历生日
生日推送通知
```

交互规则：

```Plain Text
生日卡片仅展示，不做点击跳转
```

---

## 4.5 今日生活提醒交互

提醒主题：

```Plain Text
喝水
吃饭
早睡
休息
放松
想我一下
```

交互规则：

```Plain Text
每天展示一个提醒主题
只做轻量提示
不做打卡
不做统计
不做健康管理
```

示例：

```Plain Text
今日提醒：喝水
补水有益身心健康，看到这条先喝一口。
```

---

## 4.6 今日一句交互

展示规则：

```Plain Text
每天随机 1 条
同一天保持不变
不需要用户操作
```

示例文案：

```Plain Text
普通的一天，也可以因为你变得特别。
今天也有一件值得一起完成的小事。
这里还空着，等我们慢慢填满。
```

---

## 4.7 快捷入口交互

首页底部可提供不超过 3 个快捷入口：

```Plain Text
去旅游 LIST 添加一个地方
去 Love List 添加一件小事
去筹码记一笔小手
```

交互规则：

```Plain Text
点击后切换到对应 Tab
不在首页直接展开完整列表
不在首页直接弹复杂编辑面板
```

# 5. 旅游页面交互规范

## 5.1 页面定位

旅游页面用于管理地点型愿望：

```Plain Text
我们未来想一起去的地方
```

状态只保留：


## 5.3 新增地点

```Plain Text

```

建议弹窗形式：

```Plain Text

```

字段：

```Plain Text

描述
emoji / 图标
```

新增规则：

```Plain Text

新增后即时出现在列表中
建议添加到列表顶部
```

完成反馈：


## 5.4 编辑地点

可编辑字段：

```Plain Text

描述

```

交互规则：

```Plain Text

取消后不保存修改
```

完成反馈：

```Plain Text
已更新
```

---

## 5.5 删除地点

删除必须二次确认。

确认文案：

```Plain Text
确定要删除这个想去的地方吗？
```

按钮建议：

```Plain Text

删除
```

删除完成反馈：

```Plain Text
已删除
```

---

## 5.6 点亮地点

点击“点亮”后：

```Plain Text
状态变为“已去过”
卡片出现“已点亮”标签
进度统计即时更新
```

反馈文案：

```Plain Text
这个地方已经被点亮啦。
```

---

## 5.7 取消点亮

已点亮地点允许取消点亮。

点击后：

```Plain Text

“已点亮”标签消失
进度统计即时更新
```

反馈文案：

```Plain Text
已恢复为想去。
```

# 7. 筹码页面交互规范

## 7.1 页面定位

筹码页面用于记录双方“小手余额”。

它不是惩罚系统，不是严肃记账，而是：

```Plain Text
一个用像素 IP 形象展示双方小手余额的情侣小游戏面板。
```

---

## 7.2 人物卡片交互

页面显示两张人物卡：

```Plain Text
冻冻人物卡
梨梨人物卡
```

每张卡展示：

```Plain Text
像素 IP 形象
昵称
当前资产
小愿望 / 小手图标
+1 / +5 / -1 / 清零按钮
```

---

## 7.3 筹码单位规则

筹码有两个层级：

```Plain Text
1 小手 = 1 个基础筹码
10 小手 = 1 个小愿望
```

展示规则：

```Plain Text
0–9：直接显示小手数量
10 及以上：每 10 个小手合成 1 个小愿望
剩余不足 10 的部分继续显示小手
```

示例：

```Plain Text
0 小手：暂无小手资产
1 小手：✋
5 小手：✋ ✋ ✋ ✋ ✋
10 小手：⭐
12 小手：⭐ ✋ ✋
20 小手：⭐ ⭐
27 小手：⭐ ⭐ ✋ ✋ ✋ ✋ ✋ ✋ ✋
```

如果资产超过 50，可简化为：

```Plain Text
⭐ × 5 + ✋ × 3
```

## 7.4 +1 交互

用途：普通小赌局。

点击后：

```Plain Text
对应角色小手余额 +1
图标即时更新
如达到 10，自动合成为 1 个小愿望
```

反馈文案：

```Plain Text
已记上 1 个小手。
```

## 7.5 +5 交互

点击后：

```Plain Text
对应角色小手余额 +5
图标即时更新
按 10 小手 = 1 小愿望规则自动换算
```

反馈文案：

```Plain Text
这次赢了 5 个小手，不许赖账。
```

## 7.6 -1 交互

```Plain Text
图标即时更新
```

反馈文案：

```Plain Text
已兑现 1 个小手。
```

若当前为 0：

```Plain Text
已经没有小手啦。
```

## 7.7 清零交互

按钮建议：

```Plain Text
取消
清零
```

清零后反馈：

```Plain Text
这次先放过你。
```

## 7.8 小手变星星动效


动效要求：

```Plain Text

短时间

```

# 8. 头像 / IP 交互规范

## 8.1 双层头像关系

头像系统分为：

```Plain Text
真实头像层
像素 IP 层
```

真实头像：

```Plain Text
负责现实身份识别
用于首次设置、资料识别、首页小头像、重新设置入口
```

像素 IP：

```Plain Text
负责情绪表达和专属记忆
用于首页小世界、筹码人物卡、点亮反馈、空状态
```

---

## 8.2 角色绑定

```Plain Text
冻冻 = 男生 IP，对应“我”
梨梨 = 女生 IP，对应“女朋友”
```

角色在全局保持一致，不在不同页面随意改变发色、服装和比例。

---

## 8.3 使用规则

首页：

```Plain Text
IP 作为氛围点缀
不压过在一起天数和问候文案
```

筹码页：

```Plain Text
IP 是人物卡主视觉
必须体现冻冻 / 梨梨差异
```

旅游 / Love 页：

```Plain Text
不在每张卡片中重复放头像
使用轻量像素印章、小图标、点亮徽章即可
```

# 15. 交互验收标准

## 15.1 基础验收

```Plain Text
用户首次进入能完成暗号输入和资料设置
二次进入不再要求输入暗号
底部四个 Tab 能正常切换
首页能显示正确分时问候和日期信息
旅游和 Love 能完成新增、编辑、删除、状态切换
筹码能完成 +1、+5、-1、清零
所有数据刷新后仍保留
```

---

## 15.2 体验验收

```Plain Text
用户不需要学习就能理解四个 Tab 的用途
每个页面只聚焦一个核心任务
新增 / 编辑流程不超过一个弹窗
删除和清零都有二次确认
所有状态变化都有即时反馈
没有复杂三级页面
没有只能靠长按才能发现的核心操作
```

---

## 15.3 情绪体验验收

```Plain Text
动效轻，不打扰
文案亲密但不油腻
像素元素有专属感但不喧宾夺主
筹码页面轻松可爱，不像惩罚记录
首页每天打开有一点回应感
```

---

# 16. 最终交互定义

冻梨 OS 的交互最终应做到：

```Plain Text
像 iOS 一样清晰
像小世界一样温柔
像情侣工具一样轻松
像私人礼物一样有回应
```

最重要的判断标准：

```Plain Text
用户不需要学习
用户点了马上有反馈
用户误操作可以反悔
用户每天打开不会觉得累
```
